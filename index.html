Code Maintenance and Version Control

* Source code is constantly evolving because unforeseen flaws need to be fixed and enhancements need to be incorporated; therefore, to keep a record of all changes that occur, it's important to employ some kind of version control system (VCS)

* A popular example is `Git`, but Other VCS exist like Subversion, and Mercurial

* Version control tools record all changes made to the code and who made the change, allowing you to trace and eventually recover from an unsuccessful modification.

* Furthermore, version control tools allow each developer on the team to work on a copy of the source code files without interfering with the work of other programmers. Once the new versions of source code are ready and tested, corrections or improvements made to one copy can be incorporated by other team members.

* Several online services offer storage for repositories of source code. The most popular of these services are GitHub, Gitbucketand GitLab, but the GNU project's Savannah is also worth mentioning.

Compilers and Interpreters

Interpreted languages and compiled languages are two different approaches to executing computer programs.

Machine language, also known as machine code, is a low-level programming language that is directly understood by a computer's hardware. Each type of CPU has its own unique machine language that is specific to the hardware architecture.

A compiler is a program that translates source code written in a programming language into machine language. However not all programming languages are compiled directly into machine language.

Since machine language is unique to it's hardware architecture some programs are compiled into bytecode which is a low-level instruction set that is executed by a virtual machine reffered to as a runtime, rather than directly by a computer's processor. some examples of these languages are: python, Java, C#, Ruby, and many more.

Interpreted languages are executed directly from the source code, line by line, without the need for a separate compilation step. In other words, an interpreter reads each line of code and translates it into machine instructions on-the-fly. This makes it easy to write and modify code, as changes take effect immediately. However, the downside is that interpreted languages tend to be slower than compiled languages, as each line of code needs to be translated each time it is executed.

Clients and Servers

The internet is hardware.

* In the client/server model, part of the work is done locally on the client side and part of the work is done remotely, on the server side.

A client (A computer/phone/browser/application on the phone or computer) reffers to the application utilized by the user(a person) used to request data from a server.

A server(can either be a computer or a program) stores and serves data upon request from the client.

* The client application displays an interface for the user to read, interact, or chose what data they would like to consume , but the data source and the transaction records are kept on the remote server, accessed via the network. Web applications perform this communication over the Internet, usually via the Hypertext Transfer Protocol (HTTP).

* Web application servers offer an application programming interface (API) that defines the available requests and how those requests should be made. Thus, the client constructs a request in the format defined by the API and sends it to the server, which checks for any prerequisites for the request and sends back the appropriate response.

Varieties of Web Clients

* A browser is a program that makes visible the content of HTML along with the modification of CSS, JavaScript, or other components that makes the page interactive. Browsers use a rendering engine. The rendering engine is the main component of the browser, as it is responsible for transforming HTML and other associated components into the visual and interactive elements of the interface.

* A Web Application or Web App is a program accessed by the browser, or packaged as a mobile or desktop application. Unlike the client loaded in the browser, the HTML and the necessary components for the native client to function are locally present since the installation of the application. In fact, an application that works this way is virtually identical to an HTML page (both are even likely to be rendered by the same engine).

* There are also progressive web apps (PWA), a mechanism that allows you to package web application clients for offline use‚Äîlimited to functions that do not require immediate communication with the server.

* Resources that may contain sensitive information, such as access to local files, geographic location, camera, and microphone, require explicit user authorization before the application is able to use them. This is done either through the browser or in the settings of the device for mobile ans desktop applications.

Languages Of a Client [The Frontend Stack]

* The central element of a web application client that runs on the server is the HTML document , it presents the interface elements that the browser displays in a structures way and contains the addresses for all files required for the correct presentation and operation of the client.

* HTML alone does not have much versatility to build more elaborate interfaces and does not have general-purpose programming features. For this reason, an HTML document that should function as a client application is always accompanied by one or more sets of CSS and JavaScript.

* The term document, for example, is used in JavaScript code to refer to the HTML document associated with the JavaScript code. In the context of the JavaScript language, document is a global object (of a programming class) with properties and methods that can be used to obtain information from any element on the HTML document.

Alternatives to JavaScript

* Javascript is the language of the browser, it can therefore never be replaced without replacing how browsers work. Why would the whole world replace something that works? The answer is 'we wouldn't'. That being said, it is possible to use other programming languages to create the client side of your application and compile it to JavaScript, which any browser would understand.

* Generally the preffered stack of languages used by a programmer to build the client side user interface is HTML, CSS, and JavaScript.

HTTP & APIs

* An API can be built using various programming languages such as JavaScript, Python, Java, Ruby, and many others, and can be accessed by other applications using HTTP requests. In general, an API provides a way for different software applications to interact with each other.

* APIs use HTTP (Hypertext Transfer Protocol) as a communication protocol to transfer data between the client (such as a web browser) and the server (where the API is hosted).

* When a client makes a request to an API endpoint (a specific URL), it sends an HTTP request to the server with a specific method (such as GET, POST, PUT, DELETE) and any necessary parameters or data. The server then processes the request and sends back an HTTP response containing the requested data or an error message if there was a problem.

* ‚ö†Ô∏è The misnomer client-side APIs does not mean APIs that exist on the client side, and the term `designing client APIs` does not mean building actual APIs on the client side.

* The client-side API is actually a set of functions or methods that are typically written in a programming language like JavaScript, and are used to interact with the server-side API through HTTP requests and responses. These functions or methods can be called by client-side code to send requests to the server-side API and process the responses that are received.

HTTP Requests & Responses

* HTTP requests have a few different parts that are important for working with APIs: ‚Ä¢ Method: the type of request being made (such as GET, POST, PUT, DELETE). ‚Ä¢ Endpoint: the URL where the API can be accessed and the specific resource being requested. ‚Ä¢ Headers: additional information about the request, such as the content type and authorization credentials. ‚Ä¢ Body (optional): data being sent with the request, such as form data or JSON payload.

* HTTP responses also have several important components: ‚Ä¢ Status code: a three-digit code that indicates whether the request was successful or not (such as 200 for success or 404 for not found) ‚Ä¢ Headers: additional information about the response, such as the content type and caching information ‚Ä¢ Body (optional): data being returned by the API, such as JSON or XML payload ‚Ä¢ APIs can also use other communication protocols besides HTTP, such as WebSocket or MQTT, but HTTP is the most common protocol used for web APIs.

Database Management Systems (DBMS)

* A database is a collection of data that is organized and stored in a structured manner.

* Database Management Systems (DBMS) are software programs that enable users to create, modify, and manage databases.

* Some of the common types of DBMS include relational databases, NoSQL databases, and object-oriented databases.

* Relational databases are the most commonly used type and use tables to organize data. NoSQL databases, on the other hand, use a flexible schema and do not require predefined tables. Object-oriented databases are designed to work with object-oriented programming languages and store data as objects (instances of a class).

A database connector is a software library that provides an application programming interface (API) for your program (writen in the programming language of your choice) to interact with the DBMS. This way you can use a programming language to manage data in your DBMS.
